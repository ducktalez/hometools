# iOS PWA vs. Native App — Entscheidungsdokumentation

## ⚠️ Wichtig
**Dieses File MUSS aktualisiert werden**, wenn:
- Neue iOS-Versionen (17.x+) neue APIs bringen
- Safari-Updates verhalten sich anders
- PWA-Limitations überwunden werden
- Native App-Features nötig werden

Bitte vor jeder Entscheidung „brauchen wir native App?" hier nachschlagen.

## Aktuelle Situation (2026-03-16)

**PWA-Status:** Funktionsfähig, aber mit Trade-offs
- ✅ Audio-Streaming funktioniert gut
- ✅ Video-Streaming + Miniplayer funktioniert
- ⚠️ Background-Playback + echte PWA-Installation (Home-Screen) funktioniert nicht gleichzeitig
- ❌ Offline-Download noch nicht implementiert

## iPhone-spezifische Probleme & Workarounds

### 1. `display: standalone` blockiert alle Media-APIs

**Problem:**
- `requestPictureInPicture()` → abgelehnt
- `autopictureinpicture` → ignoriert
- `requestFullscreen()` → blockiert
- Background-Media → aggressive Suspension

**Workaround:** `display: minimal-ui` verwenden
- ✅ Alle APIs funktionieren
- ❌ "Add to Home Screen" = Web-Link, keine echte PWA

**Root Cause:** iOS isoliert `standalone` WebViews komplett vom Browser

**Native App würde helfen:** Ja, native Apps haben Zugriff auf alle Media-APIs ohne diese Restrictions

---

### 2. iOS `volume`-Property ist read-only

**Problem:**
- `element.volume = 0` hat keine Wirkung
- `element.volume` gibt immer `1` zurück
- Führt zu Doppel-Audio wenn man nicht vorsichtig ist

**Workaround:** `element.muted = true/false` verwenden statt `volume`

**Native App würde helfen:** Ja, könnte `AVAudioSession` nutzen mit voller Kontrolle

---

### 3. Speicher-Quoten sind konservativ

**PWA (Home-Screen):**
- ~50 MB auf älteren iOS-Versionen
- bis zu 100-200 MB auf iOS 15+
- Mit `requestPersistentStorage()` ggf. bis 1 GB (nicht garantiert)

**Web-Link (Safari-Tab):**
- ~50 MB

**Native App:**
- `Documents/` Verzeichnis: Unbegrenzt (wird mit iCloud syncron)
- Nur limitiert durch verfügbaren Speicher
- Benutzer sieht Speicher-Nutzung in Settings

**Workaround:** Für Downloads unter 50 MB ist PWA ausreichend

**Native App würde helfen:** Ja, könnte beliebig viele Videos speichern

---

### 4. Service Worker & Background Sync

**Problem:** iOS pausiert Web-Apps aggressiv
- Service Worker wird ggf. terminiert
- Keine Background-Sync API (nur auf Android)
- Media wird beim App-Wechsel pausiert

**Workaround:** 
- `bgAudio`-Mirror als Fallback
- `visibilitychange` mit `wasPlaying`-Flag
- `autopictureinpicture` für manuelles PiP

**Native App würde helfen:** Ja, könnte Background Tasks definieren und Playback wirklich im Hintergrund halten

---

### 5. Bluetooth & Airplay

**PWA:** 
- ✅ Funktioniert (Bluetooth Headphones, Airplay)
- Aber nur wenn <audio>/<video> aktiv spielt

**Native App:**
- ✅ Bessere Integration mit Siri, Lock Screen, Control Center
- Remote Commands via `MPRemoteCommandCenter`

---

### 6. Datei-Zugriff

**PWA:**
- Nur über Fetch/HTTP
- Dateien müssen auf Server liegen
- Kein direkter Dateisystem-Zugriff

**Native App:**
- Zugriff auf `Documents/`, `Caches/`, `Temp/`
- `FileManager` API
- Share Sheet für andere Apps

**Workaround:** Alles läuft über den Server

**Native App würde helfen:** Ja, für File Picker, iCloud sync, etc.

## Timeline: Wann wird eine Native App nötig?

### 🟢 PWA reicht noch (aktuell):
1. ✅ Streaming + Background Audio
2. ✅ Miniplayer (PiP)
3. ✅ Offline-Downloads (~50 MB)
4. ✅ Basic Offline-Modus

### 🟡 PWA wird knapp:
5. ⚠️ Mehrere Videos offline (~200+ MB)
6. ⚠️ Automatische Hintergrund-Downloads
7. ⚠️ Tiefe iOS-Integration (Lock Screen, Siri)
8. ⚠️ Persistent Background Playback (länger als 10 Min)

### 🔴 Native App ist nötig:
9. ❌ Unbegrenzter Offline-Speicher
10. ❌ Echter Background Service (Downloads/Sync)
11. ❌ Lock Screen Miniatur-Vorschau
12. ❌ AirPlay/Chromecast Integration
13. ❌ Verknüpfung mit System-Mediathek
14. ❌ Native Share/Export Features

## Erkannte Patterns & Lessons Learned

### Pattern 1: "display" Mode Trade-off
**Erkannt:** 2026-03-16

| Mode | Installiert | Alle APIs | Speicher | Background |
|------|-------------|-----------|----------|------------|
| standalone | ✅ | ❌ | 50 MB | ❌ |
| minimal-ui | ⚠️ | ✅ | 50 MB | ⚠️ |
| browser | ❌ | ✅ | 50 MB | ⚠️ |

**Erkenntnis:** iOS zwingt dich zu wählen. Native App hat keine dieser Einschränkungen.

### Pattern 2: iOS `volume` ist read-only
**Erkannt:** 2026-03-16

**Fehler:** Versuch mit `volume = 0` statt `muted = true` → Doppel-Audio
**Lösung:** Immer `muted` verwenden
**Lektion:** Apple-Properties testen, nicht annehmen

### Pattern 3: visibilitychange Race Condition
**Erkannt:** 2026-03-16

**Problem:** Browser pausiert Video VOR `visibilitychange`-Event
**Lösung:** `wasPlaying`-Flag, nicht `!player.paused` prüfen
**Lektion:** Browser suspendieren Media aggressiv auf iOS

### Pattern 4: PWA Home-Screen ist kein echter App-Install
**Erkannt:** 2026-03-16

**Mythos:** PWA-Manifest = echte App
**Realität:** Nur mit `display: standalone` UND ohne `apple-mobile-web-app-capable` wird es ein Web-Link
**Lektion:** iOS PWA ist Kompromiss, keine vollwertige App

## Checkliste: Wann zur Native App migrieren?

**Entscheidungs-Kriterium:**
- [ ] PWA-Download-Feature funktioniert, aber Benutzer wollen >200MB offline
- [ ] Offline-Playback muss länger als 30 Min möglich sein
- [ ] Benutzer wollen Lock Screen Kontrollen
- [ ] Benutzer wollen automatische Background-Downloads
- [ ] iCloud-Integration gewünscht
- [ ] Siri-Integration gewünscht

**Wenn 2+ Punkte erfüllt sind:** Native App erwägen

## Native App Optionen (falls nötig)

### Option A: Swift native App (pure)
- **Kosten:** 60-80h Development + Testing
- **Vorteile:** Volle iOS-Integration, beste Performance
- **Nachteile:** Doppelter Code (Web + Swift)

### Option B: React Native / Flutter
- **Kosten:** 40-60h Development
- **Vorteile:** Code-Wiederverwendung (einige Components)
- **Nachteile:** Noch nicht so polished wie Swift

### Option C: WebView Wrapper (PWA in App)
- **Kosten:** 10-15h Development
- **Vorteile:** Minimaler Aufwand, Web-Code bleibt gleich
- **Nachteile:** Begrenzte Features (du hast kaum Zusatz-APIs)

**Empfehlung:** Bei Bedarf → Swift native App mit eigenem Backend-Call
